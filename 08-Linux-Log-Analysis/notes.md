## 1. Log Parsing & Traffic Analysis

Finding top traffic sources or broken routes (404s) requires parsing the access log. This lab demonstrates constructing custom regex and token pipelines to analyze Nginx data.

### 🛠️ Steps

1. **Missing `curl` on Ubuntu:**
   Test traffic was generated locally to populate the access log. During invocation:
   - **Rocky Linux Environment:** The `curl` utility initialized successfully natively, as RHEL-family enterprise templates bundle network diagnostic binaries by default.
   - **Ubuntu Server Environment:** Execution triggered a `command not found` boundary error due to Ubuntu's minimal default install. Resolution required manual caching:
     ```bash
     sudo apt update && sudo apt install curl -y
     ```

2. **Generating Test HTTP Requests:**
   Sent requests to generate both successful and 404 responses:

   ```bash
   # Root Target Requests (Successful Validation Hooks)
   curl -I http://localhost/

   # Missing Resource Sub-paths (404 Triggers)
   curl -I http://localhost/aa
   ```

## 🔬 Understanding the Log Line Format

To filter logs effectively, it helps to know the exact column layout. Executing `tail -n 1 /var/log/nginx/access.log` outputs a structured string split dynamically by system whitespaces:

```text
::1 - - [19/Jun/2026:17:54:14 +0300] "GET /aa HTTP/1.1" 404 0 "-" "curl/8.5.0"

```

Key columns:

- **Column 1 (`$1`):** Represents the Client IP Address (IP Address or Local Loopback Token).
- **Column 7 (`$7`):** Maps directly to the Request Path URI Target requested by the client (e.g., `/aa`).
- **Column 9 (`$9`):** Contains the explicit HTTP Status Return Code generated by the web server (e.g., `404`).

---

### 🌐 IPv4 vs IPv6 on Different Distros

Same `curl` request to `localhost` returns different formats per distro:

- **Ubuntu Resolution Mechanics (`::1`):** Ubuntu prioritizes IPv6 in `/etc/hosts`. The local loopback evaluates to the compressed IPv6 address `::1`, registering as a token in Column 1.
- **Rocky Linux Resolution Mechanics (`127.0.0.1`):** Rocky Linux defaults to IPv4. The loopback request translates cleanly to the standard IPv4 string block `127.0.0.1`.

## 🛠️ Parsing Pipelines

### 🎯 Finding the Top IP

To extract IP addresses from Column 1, the following execution pipeline is enforced:

```bash
sudo awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -nr

```

- **`awk '{print $1}'`:** Extracts only the IP address (Column 1).
- **`sort`:** Sorts so `uniq` can group duplicates.
- **`uniq -c`:** Counts occurrences of each IP.
- **`sort -nr`:** Sorts numerically, highest first.
- _Verified Result Structure:_ `53  ::1` (Ubuntu) / `53  127.0.0.1` (Rocky)

### 🎯 Counting 404 Errors by Path

To monitor brute-force routing or broken URLs without manual text scanning, we chain the search query straight to the resource index in Column 7:

```bash
sudo grep " 404 " /var/log/nginx/access.log | awk '{print $7}' | sort | uniq -c | sort -nr

```

- **`grep " 404 "`:** Keeps only lines with a 404 status.
- **`awk '{print $7}'`:** Extracts the requested path (Column 7) from those lines. (e.g., `/aa`).
- **`sort | uniq -c | sort -nr`:** Counts and sorts the paths by frequency.
- _Verified Result Structure:_ `23  /aa`
